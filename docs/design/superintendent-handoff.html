The Idea

Existing course-management software assumes conventional turf — synthetic inputs, calendar-driven maintenance, golf as the whole story. Superintendent is built for the opposite: native grasses, no synthetic chemicals, and a course that is also a working ecological system. To keep that legible, the product is organized like a farm-and-town management sim — three thematic "realms," each a board of facilities you enter and operate.

Conservation

The living land. Sensors, water, wetland, sod farm, compost, apiary, drone & weather, and the universities studying it all.

Management

The operation. Inventory, crew, schedules, events, the books, and compliance-gated suppliers.

Golf

The play. Tee sheet, pro shop, cart GPS, and a membership program wired into the conservation work.

Why a sim?

The realms aren't cosmetic. Decisions cascade between them — a booking blocks the tee sheet and staffs the crew; drone stress confirms a soil sensor; member hours fund the land.

How It Connects

The thesis is that nothing on a native course is isolated. A few of the wires worth knowing:

  • Drone stress ↔ Soil sensors — aerial NDVI patches are confirmed (or dismissed) by ground-truth moisture nodes.
  • Events → Tee Sheet + Crew + Finance — one booking cascades into course holds, staffing, and the books.
  • Members → Conservation days → Research Hub — steward volunteer hours become survey data and grant-reportable labor.
  • Composting → Finance — the closed loop shows up as fertilizer cost avoided, not just an ecological note.
  • Weather → Mowing / Water / Sod Farm — GDD and ET from one mast set seasonal decisions everywhere.
  • Suppliers → everything with soil contact — the no-synthetics rule is enforced at the purchase order.
03

Build Notes

Stack

A single-page app: React 18 + in-browser Babel, no build step. Plain .jsx files loaded as <script type="text/babel"> and composed on window. One stylesheet (sim.css) carries all design tokens. No framework, no bundler, no dependencies beyond React — in keeping with the "reliability over cleverness, no unnecessary dependencies" brief.

Theming

The whole look is driven by CSS custom properties. The light pastel-jade palette and the per-realm accent (--realm, swapped on the app frame as you change worlds) mean a tool written against the tokens themes itself correctly in any realm. Switching back to a dark theme is a one-block edit in :root.

Adding a facility tool

Four touch-points, by design:

1. write tool-yourtool.jsx → window.YourTool // use unique const names! 2. sim-shared.jsx → set the facility's tool: field 3. sim-app.jsx → register in CUSTOM + the globals list 4. Superintendent.html → add the <script> tag

One gotcha: every Babel script shares one global scope, so top-level const names must be unique across files (a STAGES / MEMBERS collision will silently pull the wrong data). Prefix tool constants.
